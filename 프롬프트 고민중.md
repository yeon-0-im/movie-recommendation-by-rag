# 리뷰 요약 프롬프트 고민 기록

`rag_pipeline/00c_summarize_reviews.py`에서 Gemini 2.0 Flash를 사용해 영화 리뷰를 구조화하는 프롬프트 설계 과정을 기록한다.

---

## 배경 — 왜 단순 요약으로는 부족한가

초기 RAG 파이프라인 테스트(`04_rag_search.py`)에서 검색 품질이 기대에 못 미쳤다.

**원인 분석**: `combined_reviews`(긍정 + 부정 리뷰 합본)를 그대로 enriched_text에 넣으면 임베딩 신호가 충돌한다.
- 예: "감동적이다"(긍정) + "지루하다"(부정) → 두 벡터가 상쇄되어 코사인 유사도가 희석됨
- "힐링 영화 추천"이라는 쿼리에 공포 영화가 섞여 들어오는 현상

**1차 개선 방향**: `pros_summary` + `cons_summary` 2개 필드로 분리 요약
- pros_summary → enriched_text에 포함 (임베딩)
- cons_summary → ChromaDB metadata에만 저장 (리랭킹 필터용)

**2차 개선 방향**: 5개 구조화 필드로 확장
- `mood_keywords`, `target_audience`, `viewing_context`, `pros_summary`, `cons_summary`
- enriched_text 구성: `장르 + 추천설명(4pillar) + 줄거리 + 분위기 + 추천대상 + 시청상황 + 리뷰`
- 사용자 쿼리 "혼자 볼 때", "데이트 영화", "힐링" 등의 상황 기반 쿼리 커버를 목표로 함

**3차 고민 (현재)**: 더 정교한 스키마로 영화의 유전자를 추출하는 프롬프트 설계

## 제안된 프롬프트 (V1 — 사용자 초안)

긍정/부정을 별도 파이프라인으로 완전 분리하고, 각각에서 구조화된 JSON을 추출하는 방식.

```
[System Role]
너는 영화의 핵심 유전자를 분석하여 정형화된 데이터(JSON)로 가공하는 구조화 엔지니어링 시스템이야.
제공된 5개의 리뷰 묶음을 분석하여, 리뷰어들이 공통으로 짚어낸 영화의 객관적 구성 요소와
주관적 정서 상태를 완벽한 MECE(상호 배타적, 전체 포괄적) 기준으로 분류하고 요약해야 해.

[Input Variables]
- 영화 제목: {title_ko}
- 파이프라인 모드: {sentiment_type} (POS 또는 NEG)
- 리뷰 데이터 뭉치:
{combined_clean_text}

[Core Instruction]
현재 파이프라인 모드는 [{sentiment_type}]야.
- 만약 모드가 POS라면: 리뷰어들이 극찬하고 매력을 느낀 '요소'와 '정서'에만 집중해.
- 만약 모드가 NEG라면: 리뷰어들이 지적한 '진입장벽', '실패 요소', '불만족스러운 정서'에만 집중해.
절대 두 감정의 요소를 타협시키거나 뭉뚱그리지 말고, 철저히 해당 모드의 관점에서만 유전자를 추출해.

[Tagging Schema & MECE Guidelines]

1. property_tags (최대 2개)
- "narrative": 스토리, 플롯, 메시지, 개연성, 대사
- "visual": 미장센, 영상미, 촬영, 색감, CG/VFX
- "audio": OST, 배경음악, 음향 효과
- "performance": 배우의 연기력, 캐릭터, 캐스팅
- "pacing": 편집 속도, 완급 조절, 지루함 여부

2. emotional_tone_weights (각각 0.0 ~ 1.0, 합이 1.0일 필요 없음)
- "exhilarating": 쾌감, 카타르시스, 유쾌한 웃음
- "suspenseful": 긴장감, 소름, 스릴, 공포
- "melancholic": 쓸쓸함, 묵직한 여운, 슬픔
- "comforting": 따뜻함, 힐링, 잔잔함, 위로
- "intellectual": 난해함, 철학적 사유, 실험성

3. focused_summary (1문장, 80자 내외)
'sentiment_type' 모드 관점에서 어떤 구체적 요소 때문에 그렇게 느꼈는지 핵심 인과관계를 요약.

[Output JSON Format]
{
  "property_tags": [],
  "emotional_tone_weights": {
    "exhilarating": 0.0,
    "suspenseful": 0.0,
    "melancholic": 0.0,
    "comforting": 0.0,
    "intellectual": 0.0
  },
  "focused_summary": ""
}
```

## 피드백 분석 — 잘 된 것과 문제점

### 잘 된 부분

- **POS/NEG 완전 분리**: 임베딩 신호 충돌 문제를 설계 수준에서 해결. 방향 자체는 옳다.
- **`emotional_tone_weights` 독립 가중치**: 합이 1.0이 아닌 것이 핵심. 영화는 "긴장감 0.8 + 카타르시스 0.7"처럼 복합적이고, 정규화하면 정보 손실.
- **`focused_summary` 인과관계 요구**: "좋았다/나빴다"가 아니라 "어떤 요소 때문에"를 강제하는 방향은 맞다.

### 문제점 1 — MECE가 실제로는 무너지는 곳

**`property_tags`:**
- `narrative`와 `pacing`이 상호 배타적이지 않음. 편집 속도 문제는 대부분 서사 구조 문제에서 비롯됨. LLM이 "지루하다"는 리뷰를 보면 둘 다 태깅하거나 임의로 하나를 고름.
- `audio`가 단독 태그로 선택될 영화가 현실적으로 거의 없음. OST 언급은 항상 narrative나 visual에 종속된 보조 역할. 독립 태그로 올라오려면 극히 음악 중심적인 영화여야 함.

**`emotional_tone_weights`:**
- `exhilarating`과 `suspenseful`이 고각성(high arousal)에서 겹침. 액션 클라이맥스는 카타르시스이면서 긴장감. 분리 기준이 없으면 LLM이 두 값을 비슷하게 매기거나 임의로 배분.
- `intellectual`은 감정 축이 아닌 인지 축. 나머지 4개(각성도-쾌감도 매트릭스)와 이질적. `melancholic + intellectual`이 동시에 높은 영화(올드보이 등)에서 두 축의 의미가 충돌.

### 문제점 2 — RAG 관점에서 핵심 필드가 사라짐

이 스키마로 바꾸면 기존 설계에서 **상황 기반 쿼리를 커버하는 필드**가 사라진다.

| 쿼리 패턴 | V1 스키마 커버 여부 |
|-----------|---------------------|
| "긴장감 있는 영화" | `suspenseful` 가중치로 커버 가능 |
| "힐링 영화" | `comforting`으로 커버 가능 |
| **"혼자 볼 때 좋은 영화"** | ❌ 커버 안 됨 |
| **"데이트 영화"** | ❌ 커버 안 됨 |
| **"취향 탈 것 같은 영화"** | ❌ 커버 안 됨 |

`focused_summary`가 이를 암묵적으로 담아줄 거라는 건 설계가 아니라 기대다.

### 문제점 3 — `emotional_tone_weights`의 활용 전략 부재

이 숫자를 ChromaDB에 저장하면 어떻게 쓸 것인가:

- **enriched_text에 텍스트로 변환** → 가중치(float)일 필요가 없고, 키워드면 충분
- **쿼리 시 감정 벡터 비교** → 사용자 쿼리에서 감정을 추출해 가중치 스코어를 따로 계산해야 함 → 파이프라인이 훨씬 복잡해짐

지금은 어느 쪽도 결정되지 않은 채 스키마만 설계된 상태. 활용 경로가 없으면 저장만 하고 쓰지 못한다.

### 문제점 4 — API 호출 2배

POS/NEG 분리 → 1,930편 × 2회 = 3,860회. 무료 티어 1,500 req/day 기준으로 기존 2일 → 약 2.6일.
품질 개선이 충분하다면 감수할 수 있지만, 하류 활용 전략이 없는 상태에서 비용만 늘어나는 위험.

## 개선 방향 — V2 설계 원칙

### 유지할 것

- **POS/NEG 분리 파이프라인** — 임베딩 신호 충돌 방지 효과가 명확
- **`emotional_tone_weights` 독립 가중치** — enriched_text에는 텍스트 키워드로 변환, metadata에는 float 보관 (장기적 필터링 확장 여지)
- **`property_tags`** — 하드웨어 분류는 유용하되, 하위 2개(`audio`, `pacing`) 재검토 필요

### 변경할 것

- **`viewing_context`를 POS 파이프라인에 명시적 복원** — "혼자 볼 때", "데이트" 등 상황 쿼리 커버는 설계로 보장해야 함
- **`focused_summary` 역할을 `viewing_context`와 분리** — focused_summary는 "왜 좋은가(인과)", viewing_context는 "언제/누구와"(상황)
- POS output: `property_tags` + `emotional_tone_weights` + `viewing_context` + `focused_summary`
- NEG output: `property_tags` + `emotional_tone_weights` + `focused_summary`

### enriched_text 최종 구성 (V2 기준)

```
장르: {tmdb_genres}
추천설명: {llm_4pillar}
줄거리: {tmdb_overview}
분위기: {emotional_tone_text}     ← emotional_tone_weights 상위 값 키워드로 변환
시청상황: {viewing_context}       ← POS 파이프라인 출력
리뷰: {focused_summary_pos}       ← POS 파이프라인 출력
```

ChromaDB metadata 전용 (임베딩 제외):
- `focused_summary_neg` — 리랭킹에서 부정 신호 필터용
- `pos_property_tags`, `neg_property_tags` — 하드웨어 속성 필터용
- `pos_tone_*`, `neg_tone_*` — 감성 가중치 flat float × 10 (ChromaDB where 절 직접 사용)

## 수정된 프롬프트 (V2)

### POS 파이프라인

```
[System Role]
너는 영화의 핵심 유전자를 분석하여 정형화된 데이터(JSON)로 가공하는 구조화 엔지니어링 시스템이야.

[Input Variables]
- 영화 제목: {title_ko}
- 장르: {genres}
- 줄거리: {overview}
- 긍정 리뷰 묶음:
{positive_reviews}

[Core Instruction]
리뷰어들이 극찬하고 매력을 느낀 '요소'와 '정서'에만 집중해.
부정적 언급은 완전히 무시하고, 긍정 관점에서만 날카롭게 유전자를 추출해.

[Tagging Schema]

1. property_tags (최대 2개 선택)
이 영화에 대한 긍정 리뷰에서 지배적으로 언급된 영화의 구성 요소를 선택해.
- "narrative": 스토리, 플롯, 메시지, 대사, 결말
- "visual": 미장센, 영상미, 촬영, 색감, CG/VFX
- "audio": OST, 음악, 음향이 핵심 강점으로 단독 언급된 경우만
- "performance": 배우 연기, 캐릭터 매력, 케미
- "pacing": 전개 속도나 편집 자체가 핵심 강점으로 단독 언급된 경우만

2. emotional_tone_weights (각각 0.0 ~ 1.0, 합산 불필요)
긍정 리뷰어들이 표현한 정서 상태를 점수화해.
- "exhilarating": 카타르시스, 유쾌한 웃음, 속 시원함, 쾌감
- "suspenseful": 긴장감, 소름, 스릴 (긍정적으로 언급된 경우)
- "melancholic": 묵직한 여운, 아름다운 슬픔, 잔잔한 고독
- "comforting": 따뜻함, 힐링, 위로, 무해함
- "intellectual": 깊은 사유 유도, 실험성, 여러 번 보게 되는 영화

3. viewing_context (1문장, 50자 내외)
긍정 리뷰를 바탕으로, 이 영화를 언제 누구와 보면 가장 좋은지 구체적인 상황을 1문장으로.
상황(혼자/연인/가족/친구)과 분위기(저녁/주말/힘들 때/기분전환)를 함께 담을 것.
예시: "혼자 조용한 밤에 감성에 젖고 싶을 때, 또는 연인과 설레는 분위기에서 보기 좋다."

4. focused_summary (1문장, 80자 내외)
단순한 감정 표현("좋았다")은 배제하고, 리뷰어들이 이 영화의 '어떤 구체적 요소' 때문에
극찬했는지 핵심 인과관계를 요약해.
예시: "원색의 감각적인 미장센과 신스웨이브 사운드가 결합하여 눈과 귀를 사로잡는 시각적 쾌감을 선사함."

출력은 순수 JSON만. 마크다운 서식이나 설명 텍스트 없이.

{
  "property_tags": [],
  "emotional_tone_weights": {
    "exhilarating": 0.0,
    "suspenseful": 0.0,
    "melancholic": 0.0,
    "comforting": 0.0,
    "intellectual": 0.0
  },
  "viewing_context": "",
  "focused_summary": ""
}
```

```
실망했는지 핵심 인과관계를 요약해.
예시: "서사의 개연성이 부족하고 인물들의 행동이 불친절하여 스토리에 몰입하기 어렵고 지루함."
```

## V2 → V3 피드백 반영 (2026-06-06)

### 피드백 1 — narrative vs pacing MECE 보완

**피드백 요지**: 두 태그의 정의가 겹친다. "후반부 플롯 전개가 늘어진다" 같은 리뷰는 이야기(narrative)의 문제인지 속도(pacing)의 문제인지 LLM이 순간적으로 양다리를 걸친다.

**평가: 수용.** 실제로 V2 정의에서 narrative에 "결말"이, pacing에 "늘어지는 전개"가 공존하는 구조였다. 경계가 흐릿하면 LLM은 일관성 없이 선택하고, 그 결과로 같은 유형의 영화가 서로 다른 태그를 받아 필터링 신뢰도가 떨어진다.

**해결 원칙 — What vs How**

| 태그 | 축 | 핵심 질문 | 대표 리뷰 표현 |
|------|-----|-----------|----------------|
| `narrative` | What | 이야기의 **내용**이 문제인가? | "개연성이 없다", "결말이 허무하다", "메시지가 없다", "대사가 진부하다" |
| `pacing` | How | 이야기의 **속도·템포**가 문제인가? | "후반부가 늘어진다", "편집이 너무 빠르다", "2시간이 순삭", "지루하게 길다" |

**판별 원칙**: 불만의 근본 원인이 **내용의 질**이면 `narrative`, **속도·흐름**이면 `pacing`.

- "결말 전개가 너무 빨라서 허무하다" → 주된 불만이 '빠름(속도)' → `pacing`
- "결말이 너무 허술하고 설명이 없다" → 주된 불만이 '내용의 부실' → `narrative`
- "갈수록 이야기가 없어지면서 늘어진다" → 내용 부재가 원인인 속도 문제 → `narrative` + `pacing` 동시 가능

---

### 피드백 2 — exhilarating vs suspenseful 교집합 보완

**피드백 요지**: 범죄 스릴러·액션 장르에서 "심장이 쫄깃하고 짜릿했다"는 긍정 리뷰를 받으면 LLM이 두 항목 모두 0.8 이상으로 채운다. 데이터 밀도가 겹쳐 분별력이 떨어진다.

**평가: 부분 수용.** 두 값이 동시에 높게 나오는 것 자체는 버그가 아니다. 긴장감이 쌓이고(suspenseful) 클라이맥스에서 해소되는(exhilarating) 영화는 실제로 두 정서를 모두 강하게 촉발한다. 이걸 억지로 분리하면 오히려 데이터가 부정확해진다.

다만 **판별 기준이 없으면 LLM이 임의로 배분**하는 문제는 맞다. "발산 에너지 vs 압박 에너지"라는 방향성 프레임을 추가하되, 두 값이 동시에 높은 케이스는 허용한다.

**해결 원칙 — 방향성 프레임**

| 감성 | 방향 | 핵심 감각 | 대표 표현 |
|------|------|-----------|-----------|
| `exhilarating` | 발산 (outward, release) | 풀리는 느낌, 터지는 느낌 | "속 시원하다", "통쾌하다", "카타르시스", "유쾌하다", "에너지 넘친다" |
| `suspenseful` | 압박 (inward, tension) | 조여드는 느낌, 쌓이는 느낌 | "심장이 쫄깃하다", "소름", "두근거린다", "손에 땀을 쥔다", "긴장된다" |

**판별 원칙**: "풀리는·터지는 느낌"이면 `exhilarating`, "조여드는·쌓이는 느낌"이면 `suspenseful`. 두 값이 모두 0.7 이상이어도 정상으로 허용.

### 추가 발견 — NEG emotional_tone_weights 정의 오류

V2 NEG 프롬프트의 emotional_tone 설명: *"기대했으나 충족되지 않은 정도"*

이 정의는 두 가지 문제가 있다.

1. **해석 방향이 불명확**: "기대했지만 없었다"인지 "기대했더니 반대로 불쾌했다"인지 모호. `melancholic`을 예로 들면:
   - "감성을 기대했는데 공감이 안 됐다" (부재)
   - "보고 나서 너무 우울하고 불쾌했다" (과잉)
   이 둘은 점수 방향이 반대인데 같은 값에 매핑된다.

2. **LLM이 실제 리뷰 텍스트에서 추론하기 어렵다**: 부정 리뷰어가 "기대했던 것"을 명시하는 경우는 드물다. "기대 충족 실패"를 추론하면 LLM이 장르 스테레오타입에 의존하게 된다.

**V3 수정**: NEG emotional_tone을 **"리뷰어가 실제로 경험한 부정적 정서의 강도"**로 재정의.

| 감성 | V2 설명 | V3 수정 설명 |
|------|---------|--------------|
| `exhilarating` | 카타르시스를 기대했으나 못 느낀 경우 | 통쾌해야 할 순간이 싱겁거나 용두사미로 끝남. 기대한 쾌감이 발생하지 않음 |
| `suspenseful` | 긴장감이 없거나 예측 가능했다는 불만 | 긴장감 없이 지루하거나 결말이 뻔하게 예측됨 |
| `melancholic` | 감성을 기대했으나 공감되지 않은 경우 | 억지 감동·신파로 감정이입 실패, 또는 보고 나서 지나치게 우울하고 불쾌함 |
| `comforting` | 힐링을 기대했으나 차갑거나 불쾌한 경우 | 보고 나서 기분이 나빠지거나 정서적으로 불쾌한 경험 |
| `intellectual` | 지나치게 난해하거나 너무 단순하다는 불만 | 불친절하게 난해하여 이해 포기, 또는 반대로 너무 단순하고 유치함 |

```
[Tagging Schema]

1. property_tags (최대 2개 선택)

  → "속 시원하다", "통쾌하다", "유쾌하다", "에너지가 넘친다"

  → "심장이 쫄깃하다", "소름 돋았다", "손에 땀을 쥔다", "긴장됐다"
  ※ 두 값이 동시에 높아도 정상 (긴장 축적 후 카타르시스 해소 구조)
- "melancholic" [내향 슬픔]: 묵직한 여운·아름다운 슬픔·잔잔한 고독·씁쓸한 감동
- "comforting" [내향 따뜻함]: 힐링·위로·잔잔함·무해함·포근함

3. viewing_context (1문장, 50자 내외)

상황(혼자/연인/가족/친구)과 분위기(저녁/주말/힘들 때/기분전환)를 함께 담을 것.
```

```
- "performance" [인물]: 어색한 연기·매력 없는 캐릭터·미스캐스팅
- "pacing" [How — 시간적 흐름]: 늘어지는 편집·과도하게 빠른 컷 전환·러닝타임 체감이 긴 경우
  → "후반부가 늘어진다", "편집이 산만하다", "길게 느껴진다"
  ※ 판별 원칙: 이야기의 '내용'이 문제면 narrative, '속도·흐름'이 문제면 pacing

2. emotional_tone_weights (각각 0.0 ~ 1.0, 합산 불필요)
부정 리뷰어들이 실제로 경험한 부정적 정서의 강도를 점수화해.
리뷰어가 '기대했던 것'이 아니라, '실제로 느낀 불만족 정서'를 기준으로 판단해.

- "exhilarating": 통쾌해야 할 순간이 싱겁거나 용두사미로 끝남. 기대한 쾌감이 발생하지 않음
  → "허무하게 끝난다", "싱겁다", "용두사미", "기대만큼 시원하지 않다"
- "suspenseful": 긴장감 없이 지루하거나 결말이 뻔하게 예측됨
  → "긴장감이 없다", "뻔하다", "예측 가능하다", "지루하다"
- "melancholic": 억지 감동·신파로 감정이입 실패, 또는 보고 나서 지나치게 우울하고 불쾌함
  → "신파다", "억지 감동", "공감이 안 된다", "보고 나서 우울하다"
- "comforting": 보고 나서 기분이 나빠지거나 정서적으로 불쾌한 경험
  → "기분이 나빠졌다", "불쾌하다", "찝찝하다", "다시 보고 싶지 않다"
- "intellectual": 불친절하게 난해하여 이해 포기, 또는 반대로 너무 단순하고 유치함
  → "무슨 말인지 모르겠다", "너무 어렵다", "유치하다", "너무 쉽다"

3. focused_summary (1문장, 80자 내외)
단순한 감정 표현("나빴다")은 배제하고, 리뷰어들이 이 영화의 '어떤 구체적 요소' 때문에
실망했는지 핵심 인과관계를 요약해.
예시: "서사의 개연성이 부족하고 인물들의 행동이 불친절하여 스토리에 몰입하기 어렵고 지루함."

출력은 순수 JSON만. 마크다운 서식이나 설명 텍스트 없이.

{
  "property_tags": [],
  "emotional_tone_weights": {
    "exhilarating": 0.0,
    "suspenseful": 0.0,
    "melancholic": 0.0,
    "comforting": 0.0,
    "intellectual": 0.0
  },
  "focused_summary": ""
}
```
