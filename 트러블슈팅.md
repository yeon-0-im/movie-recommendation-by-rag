## [2026-06-03] NoSuchWindowException — Selenium 크롤링 도중 Chrome 창 닫힘

### 증상

`python crawl/watcha_crawl.py --from-csv` 실행 중 일정 시간이 지난 후 아래 에러와 함께 전체 프로세스가 종료됨.

```
selenium.common.exceptions.NoSuchWindowException: Message: no such window: target window already closed
from unknown error: web view not found
  (Session info: chrome=148.0.7778.216)
```

스택 트레이스:
- `main()` → `crawl_titles()` → `crawl_movie()` → `search_movie_url()` → `driver.get(search_url)` 순서로 전파

### 원인

1. **직접 원인**: 장시간 크롤링 세션 도중 Chrome 브라우저 창이 예기치 않게 닫힘
   - 가능한 이유: Chrome/ChromeDriver 크래시, macOS 메모리 압박, Chrome 자체 업데이트 등
2. **전파 원인**: 기존 예외 처리가 `KeyboardInterrupt`만 잡고 있어, `NoSuchWindowException`이 위로 전파되어 배치 전체 종료

```python
# 수정 전 — KeyboardInterrupt 만 처리
try:
    reviews = crawl_movie(driver, tmdb_id, title, ...)
except KeyboardInterrupt:
    interrupted = True
    break
# NoSuchWindowException → 미처리 → crawl_titles 전체 종료
```

### 해결

**파일**: `crawl/watcha_crawl.py`

1. `NoSuchWindowException` import 추가

```python
# 수정 전
from selenium.common.exceptions import NoSuchElementException, TimeoutException

# 수정 후
from selenium.common.exceptions import NoSuchElementException, NoSuchWindowException, TimeoutException
```

2. 배치 루프에 드라이버 재시작 + 재시도 로직 추가

```python
except NoSuchWindowException:
    print(f"\n[브라우저 재시작] Chrome 창이 닫혔습니다. 드라이버를 재시작합니다...")
    try:
        driver.quit()
    except Exception:
        pass
    driver = make_driver(headless=headless)
    if not load_cookies(driver) or not is_logged_in(driver):
        print("[오류] 재로그인 실패. --save-cookie 를 다시 실행하세요.")
        interrupted = True
        break
    print(f"[브라우저 재시작] 완료. '{title}' 재시도 중...")
    try:
        reviews = crawl_movie(driver, tmdb_id, title, top_n=top_n, robots=robots)
    except Exception as e:
        print(f"  [실패] 재시도 중 오류: {e}")
        reviews = []
```

### 동작 방식 (수정 후)

- Chrome 창이 닫히면 → 기존 드라이버 종료 시도 → 새 드라이버 생성 → 쿠키 로드 + 로그인 확인
- 재로그인 성공 시 → 현재 영화 1회 재시도 후 배치 계속
- 재로그인 실패 시 (쿠키 만료) → 안전하게 중단 후 메시지 출력 (`--save-cookie` 재실행 안내)
- 재시도 중 다른 예외 발생 시 → 해당 영화 스킵, 다음 영화 계속

### 관련 스크립트

- `crawl/verify_watcha_mapping.py` — 기수집 데이터의 오매핑 여부 일괄 검증

## [2026-06-03] 영화 URL 오매핑 — 제목 단독 검색의 한계

### 증상

`review_db.csv`에 수집된 일부 리뷰가 의도한 영화가 아닌 **동명의 다른 영화** 리뷰일 가능성이 발견됨.
특히 시리즈 속편("스파이더맨 2", "에이리언 2" 등)과 동명이제("괴물" 등 흔한 제목) 영화에서 위험도가 높음.

### 원인

초기 `search_movie_url()` 구현이 검색 결과에서 **첫 번째 `/ko/contents/` 링크를 무조건 반환**했음.

```python
# 수정 전 — 아무 검증 없이 첫 번째 결과 반환
for el in links:
    href = el.get_attribute("href") or ""
    if re.search(r"/ko/contents/[A-Za-z0-9]+$", href):
        return href  # 끝
```

- "스파이더맨 2" 검색 시 검색 1위는 "스파이더맨 (2002)" → 속편이 1편 리뷰로 오수집
- "괴물" 등 흔한 제목은 의도와 다른 영화가 상위 노출될 수 있음

### 해결

**`search_movie_url()` 3단계 검증 로직으로 교체** (`crawl/watcha_crawl.py`)

```
1. title_ko 검색 → 결과 카드에서 연도 파싱
2. 연도(±1) 필터 → 후보 좁히기
3. 후보 1개 → 바로 사용
   후보 2개 이상 → 감독명 검증 (상세 페이지 추가 로드)
   후보 0개 → title_original 로 재검색 후 동일 과정
```

- 시리즈는 연도가 달라 2번에서 해결됨 (추가 비용 없음)
- 동명이제는 드물어 3번 감독 검증 비용이 거의 발생하지 않음

### 기수집 데이터 검증

**`crawl/verify_watcha_mapping.py`** 로 391편 전체를 검증 후, WRONG/SUSPECT 판정 영화만 선택적으로 재수집.

판정 기준:
- `OK` — 연도(±1), 감독 모두 일치
- `SUSPECT` — 둘 중 하나 불일치 (파싱 실패 포함 가능)
- `WRONG` — 연도, 감독 모두 불일치

```bash
python crawl/verify_watcha_mapping.py
# 결과: data/verify_result.csv
```

### 재발 방지 참고

- 쿠키가 만료된 경우 `--save-cookie` 옵션으로 재로그인 필요
- Chrome이 자주 크래시된다면 `--no-headless` 옵션으로 화면 띄워서 상태 확인
- 크롤링 속도(`DEFAULT_CRAWL_DELAY`)를 높이면 Chrome 부하 감소에 도움

## [2026-06-05] MPS 버퍼 오버플로 — BGE-M3 임베딩 중 Apple Silicon GPU 크래시

### 증상

`python rag_pipeline/02_embed_store.py` 실행 시 `Chroma.from_documents()` 단계에서 아래 에러와 함께 종료됨.

```
RuntimeError: Invalid buffer size: 14.76 GiB
```

스택 트레이스:
- `Chroma.from_documents()` → `embed_documents()` → `client.encode()` → `scaled_dot_product_attention()`

### 원인

Apple Silicon MPS에서 BGE-M3가 긴 시퀀스를 처리할 때 attention 행렬 크기가 MPS 버퍼 한도를 초과.

- BGE-M3는 최대 8,192 토큰 지원 → attention 계산이 O(n²) 메모리 필요
- `01_chunk.py`에서 최대 4,500자(≈ 6,750 토큰)로 truncate했음에도 attention 행렬이 약 14.76 GiB 요구
- MPS는 CUDA와 달리 Flash Attention을 완전 지원하지 않아 naive attention 구현 사용 → 메모리 폭증
- CPU/GPU(CUDA) 에서는 동일 조건에서 정상 동작

### 해결

**파일**: `rag_pipeline/02_embed_store.py`

1. `get_device()`에서 MPS 제외 → CUDA → CPU 순으로 변경

```python
# 수정 전
if torch.backends.mps.is_available():
    return "mps"

# 수정 후 (MPS 제외)
def get_device() -> str:
    if torch.cuda.is_available():
        return "cuda"
    return "cpu"
```

2. `batch_size=16` 추가 (추가 안전장치)

```python
encode_kwargs={"normalize_embeddings": True, "batch_size": 16}
```

### 품질 영향

없음. MPS → CPU는 속도만 다르고 임베딩 벡터 값은 동일. M 시리즈 Mac 기준 1,930편 처리에 30~60분 소요.

## [2026-06-11] 특정 영화 제외 요청이 벡터 검색에 반영되지 않음

### 증상

"이 영화 말고 다른 거 추천해줘" 또는 프론트에서 `exclude_ids`를 전달해도 제외 요청한 영화가 검색 결과에 다시 등장함. "다른 추천" 버튼을 눌렀을 때도 동일한 영화가 반복 노출됨.

### 원인

`run_converse()`가 `exclude_ids` 파라미터를 받고 있었지만, LangGraph `state`에 전달하지 않아 벡터 검색 노드까지 전파되지 않았음.

```python
# 수정 전 — exclude_ids가 state에 빠져 있음
state = {
    "query": last_user,
    "current_results": current_results,
    # exclude_ids 없음 → 검색 노드가 참조 불가
}
```

`exclude_ids`는 뷰(`views.py`)에서 `Movie.objects.exclude(id__in=exclude_ids)`로만 걸러졌기 때문에 **카드 표시 단계**에서만 숨겨졌고, **벡터 검색 자체**에는 해당 영화가 계속 포함됨. 페이지네이션(`node_paginate`)이나 재검색(`node_retry_search`) 경로에서는 필터가 전혀 적용되지 않아 제외 영화가 다시 나오는 증상 발생.

### 해결

**파일**: `backend/movies/rag.py`

1. `RAGState`에 `exclude_tmdb_ids` 필드 추가

```python
class RAGState(TypedDict):
    ...
    exclude_tmdb_ids: list  # 벡터 검색에서 제외할 tmdb_id 목록
```

2. `_build_exact_where()`에 `$nin` 조건 추가

```python
def _build_exact_where(
    tmdb_filters: dict,
    exclude_exact: dict | None = None,
    exclude_tmdb_ids: list | None = None,   # 추가
) -> dict | None:
    ...
    if exclude_tmdb_ids:
        conditions.append({"tmdb_id": {"$nin": exclude_tmdb_ids}})
```

3. `_node_vector_search` / `_node_retry_search`에서 state의 `exclude_tmdb_ids` 참조

```python
exclude_ids = state.get("exclude_tmdb_ids") or None
where = _build_exact_where(parsed["tmdb_filters"], exclude_exact or None, exclude_ids)
```

4. `run_converse()`에서 state에 전달

```python
state = {
    ...
    "exclude_tmdb_ids": list(exclude_ids) if exclude_ids else [],
}
```

### 동작 방식 (수정 후)

- 프론트에서 `exclude_ids: [101, 202]` 전달 시 → Chroma 벡터 검색의 `where` 조건에 `{"tmdb_id": {"$nin": [101, 202]}}` 적용
- 초기 검색, 재검색(retry), 페이지네이션 모든 경로에서 제외 영화가 나오지 않음
- 프론트 코드 변경 없음 — 기존에 `exclude_ids`를 body에 담아 보내던 방식 그대로 사용

## [2026-06-11] 이전 대화 맥락(제외 조건)이 후속 refine에서 사라지는 문제


### 증상

"슬픈 영화. 한국영화 빼고" 요청 후 "로맨스도 있어?" 같은 추가 필터를 입력하면, 직전에 제외했던 한국 영화가 결과에 다시 등장함. 즉, 대화 맥락으로 설정된 제외/포함 조건이 다음 턴에 유지되지 않음.

### 원인

`_node_apply_refine_filter`는 refine 요청 처리 시 이전 검색어(`prev_query`)로 DB를 재검색한 후 **현재 턴의 intent_params만** 필터로 적용했음. 이전 turnの에서 확립된 조건(예: "한국영화 빼고")이 어디에도 저장되지 않아 유실됨.

```
1턴: "슬픈 영화. 한국영화 빼고"
  → intent=search
  → _node_parse_query: exclude_exact = {tmdb_language: "ko"}
  → ChromaDB $ne 필터로 한국어 영화 제외 → 정상 작동
  → 이 조건은 parsed(ephemeral state)에만 존재, 다음 호출 시 소멸

2턴: "로맨스도 있어?"
  → intent=refine
  → _node_apply_refine_filter: exclude={}, include={genres: "로맨스"}
  → _do_search(prev_query, None, None, ...)  ← 필터 없이 전체 DB 재검색
  → _passes_refine(meta, {}, {genres: "로맨스"})  ← 로맨스 필터만 적용
  → 한국 영화 재등장
```

백엔드가 매 호출마다 fresh state로 LangGraph를 실행하기 때문에, 이전 턴의 필터 조건을 보존할 방법이 없었음.

### 해결

`session_filters`를 도입해 대화 전반에 걸쳐 제외·포함 조건을 누적. 클라이언트가 매 요청에 `session_filters`를 포함해 보내고, 서버는 업데이트된 값을 응답에 포함해 돌려줌.

**수정 파일 목록**

| 파일 | 변경 내용 |
|---|---|
| `backend/movies/rag.py` | `RAGState`에 `session_filters` 필드 추가; `_node_apply_refine_filter`에서 session과 현재 intent_params 병합; `run_converse()`에서 수신·반환 |
| `backend/movies/views.py` | 요청에서 `session_filters` 수신 → `run_converse`에 전달 → 응답에 포함 |
| `backend/movies/serializers.py` | `ChatConverseResponseSerializer`에 `session_filters` 필드 추가 |
| `frontend/src/lib/types.ts` | `ChatConverseResponse`에 `session_filters?` 타입 추가 |
| `frontend/src/lib/api.ts` | `chatConverse()`에 `sessionFilters` 파라미터 추가 |
| `frontend/src/app/page.tsx` | `sessionFiltersRef` 추가; API 호출 시 전달·갱신; 홈/새 검색 시 초기화 |

**핵심 변경 — `_node_apply_refine_filter`**

```python
# 수정 전 — 현재 턴 파라미터만 사용
filtered = [c for c in fresh if _passes_refine(c["metadata"], exclude, include)]

# 수정 후 — 이전 누적 필터 병합
session = state.get("session_filters") or {}
merged_exclude = {**session.get("exclude", {}), **exclude}  # 현재 턴 우선
merged_include = {**session.get("include", {}), **include}
filtered = [c for c in fresh if _passes_refine(c["metadata"], merged_exclude, merged_include)]
new_session = {"exclude": merged_exclude, "include": merged_include}
# → state에 new_session 저장
```

**session_filters 생명주기**

```
search 턴 → parsed.intent_exclude 추출 → session_filters 초기화 (새 검색이면 리셋)
refine 턴  → 기존 session_filters에 현재 조건 병합 → 누적
"처음부터 다시" 또는 새 탭/홈 이동 → sessionFiltersRef = {} 초기화
```

### 동작 방식 (수정 후)

1. "슬픈 영화. 한국영화 빼고" (search) → 응답에 `session_filters: {exclude: {language: "한국어"}}` 포함
2. 프론트가 `sessionFiltersRef`에 저장 → 다음 호출 시 body에 포함
3. "로맨스도 있어?" (refine) → `merged_exclude = {language: "한국어"}` + `merged_include = {genres: "로맨스"}` → 두 조건 모두 적용
4. 한국 영화 재등장 없음, 로맨스 필터도 적용됨

## [2026-06-11] refine 재검색 시 원문 사용 + 포함 조건(include) 유실

### 증상

두 가지 연관된 문제가 동시에 발생.

1. "슬픈 영화 중에 허무한 감정이 안드는 영화" 검색 후 refine 요청 시 결과가 원래 검색과 크게 달라지는 느낌 (Gemini 파싱 쿼리가 아닌 raw 원문으로 재검색됨)
2. "10대 청춘영화 일본영화로 추천해줘" 후 "애니는 싫어" refine 시, 일본 영화 조건이 사라지고 한국·미국 영화가 결과에 포함됨

### 원인

**문제 1 — raw 원문으로 재검색**

`_node_apply_refine_filter`의 재검색 쿼리가 `session_filters`가 아닌 `messages`에서 꺼낸 raw 원문을 사용했음.

```python
# 수정 전
prev_query = user_msgs[-2]  # "슬픈 영화 중에 허무한 감정이 안드는 영화" (raw 원문)
fresh = _do_search(prev_query, ...)
```

원본 검색은 Gemini가 파싱한 `semantic_query`("슬프고 텅 빈 느낌의 잔잔한 영화")로 임베딩 검색했지만, refine 재검색은 raw 원문을 사용해 임베딩 벡터가 달라지고 결과 순위가 바뀌었음.

**문제 2 — include 조건 유실**

`run_converse`에서 search 완료 후 `session_filters`를 업데이트할 때, `intent_params.include`(언어·OTT 등 포함 조건)를 현재 턴에서 새로 뽑지 않고 **이전 session의 include 값**을 그대로 복사했음.

```python
# 수정 전 — 새 include 조건 무시하고 이전 session 값 유지
updated_session = {
    "exclude": intent_exclude,
    "include": (session_filters or {}).get("include", {}),  # ← 버그
    "semantic_query": parsed.get("semantic_query", ""),
}
```

"일본영화로" 요청 시 `intent_params.include = {language: "일본어"}`가 classify_intent에서 추출됐지만 session에 저장되지 않아 다음 refine에서 유실됨.

### 해결

**파일**: `backend/movies/rag.py`

**수정 1 — `_node_apply_refine_filter`**: `session.semantic_query` 우선 사용, 없으면 raw 원문 fallback

```python
# 수정 후
raw_prev = user_msgs[-2] if len(user_msgs) >= 2 else ""
prev_query = session.get("semantic_query") or raw_prev  # Gemini 파싱 쿼리 우선
fresh = _do_search(prev_query, ...)
```

**수정 2 — `run_converse`**: 현재 턴의 `intent_params.include`를 session에 저장

```python
# 수정 후
intent_params_result = result.get("intent_params") or {}
intent_include = intent_params_result.get("include", {})  # 현재 턴 include
updated_session = {
    "exclude": intent_exclude if intent_exclude else {},
    "include": intent_include if intent_include else {},   # ← 수정
    "semantic_query": parsed.get("semantic_query", ""),
}
```

### 동작 방식 (수정 후)

```
1턴: "10대 청춘영화 일본영화로 추천해줘"
  → intent_params.include = {language: "일본어"}
  → session_filters = {
        include:        {language: "일본어"},
        exclude:        {},
        semantic_query: "10대 청춘 성장 영화",  ← Gemini 파싱 결과
    }

2턴: "애니는 싫어" (refine)
  → prev_query     = "10대 청춘 성장 영화"       ← semantic_query 재사용
  → merged_include = {language: "일본어"}        ← session에서 복원
  → merged_exclude = {genres: "애니메이션"}       ← 현재 턴
  → 일본 청춘 영화 중 애니 제외한 결과 ✓
```

## [2026-06-15] Gemini 429 Rate Limit 발생 시 UnboundLocalError → 500

### 증상

```
UnboundLocalError: cannot access local variable 'parsed' where it is not associated with a value

```

### 원인

`_node_classify_intent`의 `try` 블록 안에서만 `parsed`를 선언했는데, Gemini 429로 `except`로 빠질 경우 `parsed`가 undefined 상태로 남았음. `except` 이후 `filter_action = parsed.get(...)` 라인이 `parsed`를 참조해 `UnboundLocalError` 발생.

```python
# 수정 전
try:
    parsed = _parse_gemini_json(resp.text, {})  # try 안에서만 정의
    ...
except Exception as e:
    intent, intent_params = "search", ...      # parsed 미정의 상태

filter_action = parsed.get("filter_action", "merge")  # UnboundLocalError
```

### 해결

**파일**: `backend/movies/rag.py` — `_node_classify_intent`

`try` 블록 앞에 `parsed: dict = {}`로 초기화.

### 참고

무료 티어 기준 채팅 1회당 3회 Gemini 호출(classify_intent + parse_query + generate_pitches) → 약 6~7회 대화 후 429 도달. 429 발생 시 search fallback 처리 후 다음 요청에서 복구됨.

## [2026-06-15] 추천이유(pitch)가 표시되지 않는 현상

### 증상

영화 추천 결과에 추천이유(pitch)가 표시되지 않음. `llm_4pillar` fallback도 비어있으면 해당 영역 자체 미표시.

### 원인 1 — Ollama 미실행

pitch 생성에 `ChatOllama`를 사용했으나 Ollama 프로세스가 꺼진 상태. `generate_pitches`가 예외를 잡고 조용히 `[""] * n` 반환. 로그에 "Ollama 호출 실패"만 찍혀 원인 파악이 늦었음.

**해결**: pitch 생성을 Ollama → Gemini(`RAG_PITCH_GEMINI_MODEL`)로 교체. Ollama 코드는 품질 비교용으로 주석 보존.

### 원인 2 — pitch 파싱 정규식 MULTILINE 누락

Gemini로 교체 후에도 pitch가 간헐적으로 누락됨. `^`가 `re.MULTILINE` 없이 쓰여서 "1:" 이후 "2:", "3:" 분할 실패 → `parts` 길이 부족 → 전체 `""` 반환.

```python
# 수정 전 — MULTILINE 누락
parts = re.split(rf'(?:^|\n|(?<=\s))\s*(?:{num_pat})\s*[:：]\s*', raw)

# 수정 후 — (?m) 멀티라인, [:.：] 구분자 확장, 파싱 실패 시 원문 로깅
parts = re.split(rf'(?m)^\s*(?:{num_pat})\s*[:.：]\s*', raw)
if len(parts) < 2:
    logger.warning("[generate_pitches] 파싱 실패 — Gemini 원문: %r", raw[:200])
    return [""] * n
```

Gemini가 "애니메이션 제외"를 파싱할 때 "애니메이션" 키워드를 감지해 `include: {genres: "애니메이션"}`으로 분류. "제외" 수식어 무시.

- `CLASSIFY_INTENT_PROMPT`에 "제외" 키워드 포함 장르 제외 예시 없음
- `GEMINI_PARSER_PROMPT`의 `tmdb_filters` 제외 규칙이 약함

### 해결

**3중 방어 구조**

1. **CLASSIFY_INTENT_PROMPT**: `[제외 표현 절대 규칙]` 섹션 신설 + `"애니메이션 제외"` 등 예시 4개 추가
2. **GEMINI_PARSER_PROMPT**: `tmdb_filters`는 "포함할 조건만" 강조 + 순수 제외 쿼리 예시 3개
3. **코드 안전망**: `_node_classify_intent`에서 제외 키워드 + 장르명 패턴 감지 시 `include → exclude` 자동 교정

```python
# 정규식 모듈 레벨 상수 (매 호출 컴파일 방지)
_RE_EXCL_KW  = re.compile(r'제외|빼고|말고|빼줘|...')
_RE_GENRE_KW = re.compile(r'(애니메이션|공포|액션|...)')

# 안전망 로직
if _RE_EXCL_KW.search(q) and genre_match:
    if inc.get("genres") == matched_genre:
        exc["genres"] = inc.pop("genres")  # include → exclude 이동
```

오탐 방지: `"넷플릭스 공포영화"`(제외 키워드 없음), `"공포영화 추천"`(제외 키워드 없음) → 교정 안 함.
